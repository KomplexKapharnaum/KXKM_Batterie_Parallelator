# Phase 9: Bluetooth BLE — Design Spec

**Projet:** KXKM Batterie Parallelator (BMU)
**Date:** 2026-03-31
**Status:** Approved

---

## 1. Objectif

Monitoring + controle + configuration du BMU via Bluetooth Low Energy depuis un telephone ou une app Web Bluetooth. Profil GATT standard lisible par nRF Connect + page Web Bluetooth custom.

## 2. Profil GATT

### Service Battery (UUID: 0x1810 custom base)
- 16 characteristiques (1 par batterie, UUID 0x2A00 + offset)
- Format : struct packed { float voltage_mv; float current_a; uint8_t state; float ah_discharge; float ah_charge; uint8_t nb_switch; }
- Properties : READ + NOTIFY (push toutes les 1s)
- Pas de pairing requis pour la lecture

### Service System (UUID custom)
- Characteristique Firmware : string "0.4.0" (READ)
- Characteristique Heap : uint32 free bytes (READ + NOTIFY 10s)
- Characteristique Uptime : uint32 seconds (READ)
- Characteristique WiFi : string IP ou "disconnected" (READ)
- Characteristique Topology : uint8 nb_ina, uint8 nb_tca, bool valid (READ)
- Characteristique Solar : struct VE.Direct data (READ + NOTIFY si connecte)

### Service Control (UUID custom) — PAIRING REQUIS
- Characteristique Switch : write { uint8_t battery_idx; uint8_t on_off; }
  → appelle `bmu_protection_web_switch()`
  → response dans une characteristique status
- Characteristique Reset : write { uint8_t battery_idx; }
  → appelle `bmu_protection_reset_switch_count()`
- Characteristique Config : write { uint16_t min_mv; uint16_t max_mv; uint16_t max_ma; uint16_t diff_mv; }
  → sauvegarde en NVS, applique au runtime

### Securite
- Monitoring (reads) : sans pairing, ouvert
- Controle (writes) : pairing + bonding requis (MITM protection optionnel)
- Bonding : cle stockee en NVS, pas de re-pairing apres premiere connexion
- Max 3 connexions simultanées

## 3. Stack BLE

- **NimBLE** (pas Bluedroid) — plus leger (~60 KB flash vs ~200 KB)
- `CONFIG_BT_NIMBLE_ENABLED=y` dans sdkconfig
- `esp_nimble_hci` + `nimble_host` + `nimble_svc_gap` + `nimble_svc_gatt`

## 4. Web Bluetooth

- Page HTML dans SPIFFS : `/ble.html`
- Accessible via WiFi (`http://<ip>/ble.html`) ou ouvrable localement
- API Web Bluetooth (Chrome Android/desktop, Edge — PAS Safari iOS)
- Fonctionnalites :
  - Scan + connexion au BMU
  - Grille batteries live (via NOTIFY)
  - Boutons switch ON/OFF
  - Page config seuils
  - Donnees solaire VE.Direct

## 5. Composant ESP-IDF

```
firmware-idf/components/bmu_ble/
├── CMakeLists.txt
├── Kconfig                     # Enable BLE, device name, max connections
├── include/bmu_ble.h           # Init + GATT service registration
├── bmu_ble.cpp                 # NimBLE init + GAP + advertising
├── bmu_ble_battery_svc.cpp     # Battery service (16 chars)
├── bmu_ble_system_svc.cpp      # System service
└── bmu_ble_control_svc.cpp     # Control service (pairing required)
```

### API
```cpp
esp_err_t bmu_ble_init(bmu_protection_ctx_t *prot,
                        bmu_battery_manager_t *mgr,
                        uint8_t nb_ina);
bool bmu_ble_is_connected(void);
int bmu_ble_connected_count(void);
```

### Kconfig
```
menu "BMU Bluetooth"
    config BMU_BLE_ENABLED
        bool "Enable Bluetooth BLE"
        default n
    config BMU_BLE_DEVICE_NAME
        string "BLE device name"
        default "KXKM-BMU"
        depends on BMU_BLE_ENABLED
    config BMU_BLE_MAX_CONNECTIONS
        int "Max simultaneous connections"
        default 3
        depends on BMU_BLE_ENABLED
endmenu
```

## 6. Integration

- `main.cpp` : ajouter `bmu_ble_init()` apres protection init, gated par `CONFIG_BMU_BLE_ENABLED`
- Pas de conflit avec WiFi : ESP32-S3 supporte WiFi + BLE simultanément
- Les writes BLE control passent par `bmu_protection_web_switch()` (meme chemin securise que le web)
- Impact memoire : NimBLE ~60 KB flash + ~10 KB RAM — reste dans le budget OTA 2 MB
