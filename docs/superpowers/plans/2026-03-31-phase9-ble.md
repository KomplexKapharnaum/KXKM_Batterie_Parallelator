# Phase 9: Bluetooth BLE — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add BLE GATT services for battery monitoring, system info, and control with NimBLE + Web Bluetooth page.

**Architecture:** Single new component `bmu_ble` exposing three GATT services: Battery (16 characteristics, READ+NOTIFY 1s), System (firmware/heap/uptime/wifi/topology/solar, READ+NOTIFY 10s), and Control (switch/reset/config writes, pairing required). NimBLE stack (~60 KB flash, ~10 KB RAM). Web Bluetooth HTML page served via SPIFFS for Chrome/Edge clients. Control writes reuse existing `bmu_protection_web_switch()` and `bmu_protection_reset_switch_count()` — same secured path as HTTP mutations. Config writes persist to NVS.

**Tech Stack:** ESP-IDF v5.4, NimBLE, GATT, Web Bluetooth API

**Branch:** `feat/phase9-ble`

## UUIDs

Custom 128-bit base: `4b584b4d-xxxx-4b4d-424d-55424c450000` (KXKM-BMU encoded in hex).

| Service / Characteristic | UUID suffix |
|--------------------------|-------------|
| Battery Service          | `0001`      |
| Battery Char [0..15]     | `0010` .. `001F` |
| System Service           | `0002`      |
| System Firmware          | `0020`      |
| System Heap              | `0021`      |
| System Uptime            | `0022`      |
| System WiFi IP           | `0023`      |
| System Topology          | `0024`      |
| System Solar             | `0025`      |
| Control Service          | `0003`      |
| Control Switch           | `0030`      |
| Control Reset            | `0031`      |
| Control Config           | `0032`      |

  #pragma once
  #include "bmu_protection.h"
  #include "bmu_battery_manager.h"

  #ifdef __cplusplus
  extern "C" {
  #endif

  /**
   * @brief Initialise NimBLE, enregistre les 3 services GATT, demarre l'advertising.
   *        Doit etre appele apres bmu_protection_init et bmu_battery_manager_init.
   *

- [ ] Create `bmu_ble.cpp` implementing:
  - Static globals: `s_prot`, `s_mgr`, `s_nb_ina`, `s_connected_count` (atomic)
  - `bmu_ble_init()`:
    1. Store context pointers in statics
    2. Call `nimble_port_init()`
    3. Configure GAP: device name from `CONFIG_BMU_BLE_DEVICE_NAME`, appearance = Generic
    4. Register battery service (from `bmu_ble_battery_svc.cpp`)
    5. Register system service (from `bmu_ble_system_svc.cpp`)
    6. Register control service (from `bmu_ble_control_svc.cpp`)
    7. Configure security manager for bonding:
       - `ble_hs_cfg.sm_io_cap = BLE_SM_IO_CAP_NO_IO` (Just Works pairing)
       - `ble_hs_cfg.sm_bonding = 1`
       - `ble_hs_cfg.sm_our_key_dist = BLE_SM_PAIR_KEY_DIST_ENC | BLE_SM_PAIR_KEY_DIST_ID`
       - `ble_hs_cfg.sm_their_key_dist = BLE_SM_PAIR_KEY_DIST_ENC | BLE_SM_PAIR_KEY_DIST_ID`
       - `ble_hs_cfg.sm_sc = 1` (Secure Connections)
    8. Start NimBLE host task via `nimble_port_freertos_init()`
  - GAP event callback `ble_gap_event_handler()`:
    - `BLE_GAP_EVENT_CONNECT`: increment `s_connected_count`, log
    - `BLE_GAP_EVENT_DISCONNECT`: decrement `s_connected_count`, restart advertising
    - `BLE_GAP_EVENT_ADV_COMPLETE`: restart advertising
    - `BLE_GAP_EVENT_ENC_CHANGE`: log encryption status
    - `BLE_GAP_EVENT_REPEAT_PAIRING`: delete old bond, allow re-pair
  - `start_advertising()`:
    - Flags: `BLE_HS_ADV_F_DISC_GEN | BLE_HS_ADV_F_BREDR_UNSUP`
    - Include device name in advertising data
    - Include 128-bit Battery Service UUID in scan response
    - Connection interval: 30–60 ms
    - Advertising indefinitely (duration = `BLE_HS_FOREVER`)
    - Max connections gated by `CONFIG_BMU_BLE_MAX_CONNECTIONS`
  - `on_sync()` callback (called when host syncs): print address, call `start_advertising()`
  - `on_reset()` callback: log reason, no-op
  - `bmu_ble_is_connected()`: return `s_connected_count > 0`
  - `bmu_ble_connected_count()`: return `s_connected_count`
  - Internal getters used by service files: `bmu_ble_get_prot()`, `bmu_ble_get_mgr()`, `bmu_ble_get_nb_ina()` — declared in a private header or as extern in the .cpp files

### Task 2: Battery GATT service (16 characteristics, READ + NOTIFY)

**Files:**
- Create: `firmware-idf/components/bmu_ble/bmu_ble_battery_svc.cpp`

**Steps:**

- [ ] Define packed battery characteristic struct:
  ```cpp
  typedef struct __attribute__((packed)) {
      int32_t  voltage_mv;    // from bmu_protection_get_voltage() (float → int32)
      int32_t  current_ma;    // from bmu_ina237_read_current() (A → mA, float → int32)
      uint8_t  state;         // bmu_battery_state_t enum value
      int32_t  ah_discharge;  // from bmu_battery_manager (float * 1000 → mAh as int32)
      int32_t  ah_charge;     // from bmu_battery_manager (float * 1000 → mAh as int32)
      uint8_t  nb_switch;     // from prot->nb_switch[i]
  } ble_battery_char_t;      // 15 bytes total
  ```
  Note: integer encoding avoids float endianness issues across BLE clients.

- [ ] Define 16 characteristic definitions using the UUID table (0x0010..0x001F), each with:
  - `BLE_GATT_CHR_F_READ | BLE_GATT_CHR_F_NOTIFY`
  - Access callback: `battery_chr_access_cb` with `arg` = battery index
  - No security flags (open read per spec)

- [ ] Implement `battery_chr_access_cb()`:
  - On `BLE_GATT_ACCESS_OP_READ_CHR`:
    1. Extract battery index from `arg`
    2. Validate index < `bmu_ble_get_nb_ina()`
    3. Read voltage from `bmu_protection_get_voltage()`
    4. Read current from `bmu_ina237_read_current()`
    5. Read state from `bmu_protection_get_state()`
    6. Read Ah from `bmu_battery_manager_get_ah_discharge/charge()`
    7. Read nb_switch from `prot->nb_switch[i]` (mutex-protected)
    8. Pack into `ble_battery_char_t`, append to `os_mbuf` via `os_mbuf_append()`
  - Return `BLE_ATT_ERR_INVALID_HANDLE` for invalid index

- [ ] Implement 1-second notify timer:
  - Create `esp_timer` in service registration function
  - Timer callback iterates all `nb_ina` batteries:
    1. Build `ble_battery_char_t` for each
    2. Call `ble_gatts_notify_custom()` with the characteristic value handle
    3. Skip if no subscribers (check `ble_gap_conn_find()`)
  - Timer started when first client subscribes (CCCD write), stopped when last unsubscribes
  - Use static array `uint16_t s_battery_val_handles[16]` populated during service registration

- [ ] Register service via `ble_gatts_count_cfg()` + `ble_gatts_add_svcs()`, called from `bmu_ble_init()`
  - Export: `const struct ble_gatt_svc_def *bmu_ble_battery_svc_def(void)`

**Verification:**
- nRF Connect: read each battery characteristic, verify voltage/current match serial log
- nRF Connect: enable notifications on battery 0, verify 1s update cadence
- Read all 16 characteristics in < 200 ms (no I2C contention stalls)

### Task 3: System GATT service

**Files:**
- Create: `firmware-idf/components/bmu_ble/bmu_ble_system_svc.cpp`

**Steps:**

- [ ] Define 6 characteristics with UUIDs 0x0020..0x0025:

  | Char | UUID suffix | Type | Properties | Content |
  |------|-------------|------|------------|---------|
  | Firmware | 0x0020 | UTF-8 string | READ | `CONFIG_APP_PROJECT_VER` ("0.4.0") |
  | Heap | 0x0021 | uint32_t | READ + NOTIFY | `esp_get_free_heap_size()` |
  | Uptime | 0x0022 | uint32_t | READ | `esp_timer_get_time() / 1000000` (seconds) |
  | WiFi IP | 0x0023 | UTF-8 string | READ | `bmu_wifi_get_ip()` or "disconnected" |
  | Topology | 0x0024 | packed struct | READ | `{nb_ina, nb_tca, valid}` (3 bytes) |
  | Solar | 0x0025 | packed struct | READ + NOTIFY | VE.Direct data subset |

- [ ] Define solar packed struct:
  ```cpp
  typedef struct __attribute__((packed)) {
      int16_t  battery_voltage_mv;   // V * 1000
      int16_t  battery_current_ma;   // A * 1000
      uint16_t panel_voltage_mv;     // V * 1000
      uint16_t panel_power_w;
      uint8_t  charge_state;         // CS enum
      uint8_t  error_code;           // ERR
      uint32_t yield_today_wh;
      uint8_t  valid;                // 0 or 1
  } ble_solar_char_t;               // 13 bytes
  ```

- [ ] Implement `system_chr_access_cb()`:
  - Switch on characteristic UUID suffix to determine which data to return
  - Firmware: `os_mbuf_append()` with version string
  - Heap: pack `esp_get_free_heap_size()` as little-endian uint32
  - Uptime: pack `(uint32_t)(esp_timer_get_time() / 1000000ULL)`
  - WiFi IP: call `bmu_wifi_get_ip()`, fall back to "disconnected"
  - Topology: pack `{prot->nb_ina, prot->nb_tca, topology_valid}` using protection context
  - Solar: call `bmu_vedirect_get_data()`, pack into `ble_solar_char_t`, set valid=0 if `!bmu_vedirect_is_connected()`

- [ ] Implement 10-second notify timer:
  - `esp_timer` callback notifies Heap and Solar characteristics
  - Same subscribe/unsubscribe gating as battery service

- [ ] Export: `const struct ble_gatt_svc_def *bmu_ble_system_svc_def(void)`

**Verification:**
- nRF Connect: read firmware char, verify "0.4.0"
- nRF Connect: read heap char, verify value matches serial STATUS log
- nRF Connect: read solar char, verify valid=0 if no Victron connected
- Enable notify on heap, verify 10s cadence

### Task 4: Control GATT service (pairing required)

**Files:**
- Create: `firmware-idf/components/bmu_ble/bmu_ble_control_svc.cpp`

**Steps:**

- [ ] Define 3 characteristics with UUIDs 0x0030..0x0032, all with **write + encryption required**:

  | Char | UUID suffix | Properties | Payload |
  |------|-------------|------------|---------|
  | Switch | 0x0030 | WRITE + ENC_REQ | `{uint8_t battery_idx, uint8_t on_off}` (2 bytes) |
  | Reset | 0x0031 | WRITE + ENC_REQ | `{uint8_t battery_idx}` (1 byte) |
  | Config | 0x0032 | WRITE + ENC_REQ | `{uint16_t min_mv, uint16_t max_mv, uint16_t max_ma, uint16_t diff_mv}` (8 bytes LE) |

  Flags for each: `BLE_GATT_CHR_F_WRITE | BLE_GATT_CHR_F_WRITE_ENC`

- [ ] Add a status/response characteristic (UUID 0x0033, READ + NOTIFY):
  ```cpp
  typedef struct __attribute__((packed)) {
      uint8_t  last_cmd;       // 0=switch, 1=reset, 2=config
      uint8_t  battery_idx;    // 0xFF if N/A (config)
      uint8_t  result;         // 0=OK, 1=invalid_idx, 2=locked, 3=error
  } ble_control_status_t;     // 3 bytes
  ```

- [ ] Implement `control_chr_access_cb()`:
  - On `BLE_GATT_ACCESS_OP_WRITE_CHR`, switch on UUID:
  - **Switch** (0x0030):
    1. Validate payload length == 2
    2. Extract `battery_idx` and `on_off`
    3. Validate `battery_idx < bmu_ble_get_nb_ina()`
    4. Call `bmu_protection_web_switch(prot, battery_idx, on_off)`
    5. Map return: `ESP_OK` → result=0, `ESP_ERR_INVALID_ARG` → result=1, `ESP_ERR_NOT_ALLOWED` → result=2, else → result=3
    6. Update status characteristic, notify
  - **Reset** (0x0031):
    1. Validate payload length == 1
    2. Extract `battery_idx`, validate range
    3. Call `bmu_protection_reset_switch_count(prot, battery_idx)`
    4. Update status characteristic, notify
  - **Config** (0x0032):
    1. Validate payload length == 8
    2. Extract 4 uint16_t values (little-endian)
    3. Validate ranges: min_mv in [20000, 30000], max_mv in [25000, 35000], max_ma in [1000, 50000], diff_mv in [100, 5000]
    4. Save to NVS namespace "bmu_config": keys `min_mv`, `max_mv`, `max_ma`, `diff_mv`
    5. Log new values, update status characteristic
    6. Note: runtime application of config requires restart (NVS-backed Kconfig values are read at boot)

- [ ] Pairing enforcement:
  - The `BLE_GATT_CHR_F_WRITE_ENC` flag causes NimBLE to automatically initiate pairing when an unauthenticated client attempts a write
  - Bonding keys stored in NVS via `CONFIG_BT_NIMBLE_NVS_PERSIST=y`
  - On `BLE_GAP_EVENT_REPEAT_PAIRING` in GAP handler: delete old bond, accept new pairing

- [ ] Export: `const struct ble_gatt_svc_def *bmu_ble_control_svc_def(void)`

**Verification:**
- nRF Connect: attempt write to Switch char without pairing → pairing dialog triggered
- nRF Connect: write `{0x00, 0x01}` to Switch char after pairing → battery 0 switches ON, status notifies result=0
- nRF Connect: write `{0xFF}` to Reset char → status notifies result=1 (invalid index)
- nRF Connect: write 8-byte config → verify NVS persistence with `nvs_flash` dump
- Reconnect after bonding: no re-pairing required
